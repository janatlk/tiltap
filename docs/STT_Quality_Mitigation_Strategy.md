# Стратегия повышения качества STT для таджикского и кыргызского языков

**Проект:** TilTap  
**Дата:** 03.07.2026  
**Автор:** Жанат Жанузаков (подготовлено с помощью Kimi Code CLI)  

## 1. Исходные условия и ограничения

Из ответов Заказчика и реального состояния сервера (Hetzner CPX22, 2 vCPU, 4 GB RAM, без GPU):

| Параметр | Значение |
|----------|----------|
| STT-модели | **Только локальные open-source** — облачные STT API не используются |
| Бюджет на разметку | **Отсутствует** — нельзя заказывать ручную транскрипцию |
| Доминирующий контент | **Длинные YouTube/медиа** (подкасты, интервью, ролики) |
| Сервер | Hetzner CPX22, 2 vCPU, 4 GB RAM, без GPU, Docker-лимит для STT ≈ 2.5 GB |
| Текущие модели на сервере | `vosk_small_ky`, `vosk_small_uz`, `rubai_uz`, `whisper_distil` |

Известные результаты проекта (по Obsidian-заметкам):

| Язык | Модель / тест | Результат |
|------|---------------|-----------|
| Кыргызский | Vosk large на 1-мин YouTube | **100% / 100%** (char / word) |
| Таджикский | Fine-tuned Whisper на 60-с интервью | **93.8% / 92.7%** |
| Узбекский | Vosk small на 67-с YouTube | **88.0% / 42.9%** |
| Phrasebook (короткие фразы) | Все модели | ~20% char — слабое место |

**Вывод:** на длинных чистых записях качество уже высокое. Проблемы — в коротких изолированных фразах, шуме, музыке, незнакомых именах/терминах, код-свитчинге и нехватке RAM для загрузки лучших моделей.

## 2. Где именно теряется качество

1. **Короткие фразы с паузами** — Whisper с `condition_on_previous_text` «галлюцинирует» повторы и вставки.
2. **Шум, музыка, эхо** — ухудшают все модели, особенно Vosk small.
3. **Код-свитчинг (русские вставки)** — модель переключается на неправильный язык.
4. **Имена собственные, топонимы, даты, специфическая лексика** — out-of-vocabulary ошибки.
5. **Недостаток RAM на сервере** — в production не загружается лучшая кыргызская модель (`vosk-model-ky-0.42`), используется small-версия.
6. **Нет автоматического контроля качества** — результат с низким confidence не перезапускается альтернативной моделью.

## 3. Матрица решений

| № | Подход | Ожидаемый прирост качества | Стоимость | Сложность | Скорость | Риски |
|---|--------|---------------------------|-----------|-----------|----------|-------|
| 1 | **Апгрейд сервера до CPX32 (8 GB RAM)** и загрузка лучших моделей | Высокий для ky (+10–50% относительного снижения ошибок), средний для tg/uz | ~10–15 $/мес | Низкая | 1–3 дня | Узкое горло — CPU, скорость останется невысокой |
| 2 | **Дообучение (fine-tuning) Whisper / Wav2Vec2** на публичных датасетах (Common Voice, FLEURS, YouTube captions) | Средний-высокий (+5–20% char) | Облачный GPU для обучения (разовые ~20–100 $) | Высокая | 1–3 мес | Нужны данные/псевдо-разметка; без GPU на сервере inference медленный |
| 3 | **LoRA / adapters** для Whisper на целевых доменах | Средний (+3–10% char) | Низкая (обучение на GPU ~10–30 $) | Средняя | 2–6 нед | Меньше требует данных, чем полный fine-tune |
| 4 | **VAD + сегментация аудио** и транскрипция по сегментам без previous-text conditioning | Высокий для phrasebook (+20–40% char), средний для медиа (+5–15%) | Бесплатно | Средняя | 2–3 нед | Возможны ошибки на границах сегментов |
| 5 | **Денойзинг и аудио-энхансмент** (RNNoise, Speex, ffmpeg нормализация, high-pass) | Средний (+5–15% char на шумных записях) | Бесплатно | Низкая | 1–2 нед | Агрессивный денойзинг может «съесть» речь |
| 6 | **Confidence-based quality gate + fallback chain** | Средний (+5–20% общий) | Бесплатно | Средняя | 2–4 нед | Увеличивает время обработки |
| 7 | **N-best rescoring** с языковой моделью / LLM | Средний (+3–10% char) | LLM API (OpenAI/Groq) или локальная маленькая LLM | Средняя | 2–4 нед | LLM может переиначить смысл |
| 8 | **Lexicon biasing / hotword boosting** для Vosk + контекстные промпты для Whisper | Средний (+5–15% на доменной лексике) | Бесплатно | Средняя | 2–3 нед | Требует постоянной поддержки словарей |
| 9 | **RAG / внешние словари** имен, топонимов, дат, терминов | Средний (+5–10% на именах/датах) | Бесплатно | Низкая | 1–2 нед | Не исправляет фонетические ошибки |
| 10 | **Псевдо-разметка (self-training)** неразмеченных аудио с фильтрацией по confidence | Средний (+5–15% при наличии данных) | Вычислительные ресурсы | Высокая | 2–4 мес | Качество зависит от уверенности baseline-модели |
| 11 | **Синтетические данные** через TTS для коротких фраз | Средний для phrasebook (+5–15%) | TTS-сервис / локальная TTS-модель | Средняя | 2–6 нед | TTS может звучать неестественно; качество TTS для ky/tg неизвестно |
| 12 | **Диаризация** для многоspeaker-контента | Средний (+5–10% на интервью/панелях) | pyannote-модель, CPU/RAM | Средняя | 2–4 нед | Дополнительная модель увеличивает нагрузку |
| 13 | **Обработка код-свитчинга** (авто + forced Russian) | Средний (+5–10%) | Бесплатно | Средняя | 2–3 нед | Уже есть `multi`-режим; требует доработки |
| 14 | **Human-in-the-loop** — сбор исправлений пользователей | Постепенный (+1–3% в месяц накопительно) | Только разработка UX | Низкая | 2–4 нед | Нужна политика приватности и согласие |
| 15 | **Замена Whisper distil на large-v3-turbo / large-v3** как fallback | Средний (+5–15% для tg/ru/en) | Больше RAM/CPU | Низкая | 1–2 нед | Медленнее, требует больше памяти |

## 4. Стратегия по языкам

### 4.1. Кыргызский (ky)

**Быстрые результаты (0–4 недели):**
1. **Апгрейд RAM до 8 GB** и загрузка `vosk-model-ky-0.42` — это главный рычаг, поскольку small-модель сильно уступает large на длинных записях.
2. **VAD + сегментация** для разбиения длинного аудио на речевые сегменты.
3. **Денойзинг** через RNNoise/speex перед Vosk.
4. **Lexicon biasing** для кыргызских имён, топонимов, политических/общественных терминов.
5. **RAG-словарь** кыргызских имен собственных и дат.

**Среднесрочные (1–3 месяца):**
- Ensemble `Vosk large` + `Vosk small` с confidence-based выбором.
- N-best rescoring для кыргызских гипотез.
- Сбор публичных данных (Common Voice Kyrgyz) и псевдо-разметка YouTube-аудио для адаптации языковой модели.

### 4.2. Таджикский (tg)

**Быстрые результаты (0–4 недели):**
1. Убедиться, что `whisper-tajik-finetuned-ct2` загружена в production (сейчас health-check показывает только `whisper_distil`).
2. **Отключение `condition_on_previous_text`** для коротких фраз и фрагментов с музыкой/паузами.
3. **Улучшение cleanup/post-processing** — уже есть, но можно расширить словарь named entities и дат.
4. **Контекстные промпты** для Whisper с подсказкой о домене (новости, интервью).

**Среднесрочные (1–3 месяца):**
- Fallback на `whisper-large-v3-turbo` при низком confidence fine-tuned модели.
- VAD + сегментация для phrasebook/VM.
- Дообучение fine-tuned модели на FLEURS Tajik + Common Voice Tajik + псевдо-разметке с помощью облачного GPU.

### 4.3. Общие инженерные меры

1. **Confidence-based quality gate:** после STT проверять средний confidence, длину результата, наличие повторов/галлюцинаций. При срабатывании — fallback на следующую модель в цепочке.
2. **Аудио-конвейер:** ffmpeg → resample 16 kHz mono → denoise → VAD → normalization → STT.
3. **Пост-обработка LLM:** консервативная корректировка пунктуации, имён, дат; без изменения содержания.
4. **Мониторинг:** логировать модель, confidence, fallback, время, язык, источник — для приоритизации следующих улучшений.

## 5. Рекомендуемая дорожная карта (12 месяцев)

### Месяц 1 — «Быстрые победы»
- [ ] Апгрейд Hetzner CPX22 → CPX32 (8 GB RAM).
- [ ] Загрузка `vosk-model-ky-0.42` и `whisper-tajik-finetuned-ct2` в production.
- [ ] Внедрить VAD + сегментацию с отключением previous-text для коротких фраз.
- [ ] Добавить RNNoise/speex денойзинг в аудио-конвейер.
- [ ] Расширить rule-based cleanup и словари для ky/tg.

**Ожидаемый эффект:** заметное улучшение на шумных длинных медиа и phrasebook; ky получит наибольший прирост.

### Месяцы 2–3 — «Контроль качества и fallback»
- [ ] Реализовать confidence-based quality gate и fallback chain.
- [ ] N-best rescoring для гипотез.
- [ ] Lexicon biasing / hotword boosting для Vosk.
- [ ] Обработка код-свитчинга (auto + forced Russian).
- [ ] Диаризация для многоspeaker-контента.

**Ожидаемый эффект:** снижение количества явно мусорных транскрипций и улучшение доменной лексики.

### Месяцы 4–6 — «Данные без разметки»
- [ ] Псевдо-разметка неразмеченных YouTube-аудио для ky/tg с фильтрацией по confidence.
- [ ] Сбор публичных датасетов Common Voice / FLEURS / Mozilla Common Voice Tajik/Kyrgyz.
- [ ] LoRA fine-tuning Whisper на собранных данных через арендованный GPU (обучение вне Hetzner, inference — локально).
- [ ] Генерация синтетических phrasebook-фраз через TTS.

**Ожидаемый эффект:** +5–15% char на целевых доменах без затрат на ручную разметку.

### Месяцы 7–12 — «Накопление и масштабирование»
- [ ] Human-in-the-loop: пользовательские исправления → обновление словарей и периодический retrain.
- [ ] Оптимизация inference: int8-квантизация, batching, кэширование моделей.
- [ ] Расширение lexicon/RAG на новые домены (политика, медицина, образование).
- [ ] При достижении лимитов CPU — переход на сервер с GPU или выделенный STT-сервер.

## 6. Что не рекомендуется

| Подход | Почему не в приоритете |
|--------|------------------------|
| Ручная разметка данных | Заказчик исключил бюджет на разметку |
| Полное обучение STT с нуля | Требует сотен/тысяч часов аудио и GPU-кластер |
| Использование коммерческих STT API | Запрещено условием «только локальные модели» |
| Слепое увеличение моделей без VAD/денойза | На шумных данных прирост будет минимальным |

## 8. Итоговая рекомендация

**Оптимальная стратегия для ближайших 12 месяцев:**

1. **Апгрейд RAM + загрузка лучших моделей** — даёт самый быстрый и заметный прирост для кыргызского.
2. **VAD/сегментация + денойзинг + отключение previous-text** — решает главную проблему phrasebook и шумных медиа.
3. **Confidence-based fallback + N-best LLM rescoring** — снижает количество явных ошибок.
4. **Lexicon biasing + RAG-словари** — повышает точность на именах и терминах без затрат на разметку.
5. **Псевдо-разметка + LoRA fine-tuning на арендованном GPU** — среднесрочный прирост без ручной разметки.
6. **Human-in-the-loop** — долгосрочное накопление улучшений.

Эта комбинация не требует коммерческих STT API и ручной разметки, реализуется в течение года и адресует конкретные слабые места, выявленные в проекте.
