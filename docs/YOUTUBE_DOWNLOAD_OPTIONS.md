# YouTube: как автоматизировать скачивание с бэкенда

> Анализ от 2026-07-24. Все замеры сделаны с боевого сервера `tiltab-cx43-hel2`
> (`95.216.169.56`). Пересматривать при смене хостинга или egress-IP.

## 1. Что измерено (факты, не теория)

| Проверка | Результат |
|---|---|
| yt-dlp 2026.06.09, клиент `default` | ❌ Sign in to confirm you're not a bot |
| клиенты `tv`, `ios`, `mweb`, `web_safari`, `android_vr` | ❌ все до одного, тот же ответ |
| обычная страница `youtube.com/watch` | ✅ HTTP 200 |
| Cobalt `liubquanti` (чужой IP) | ✅ отдаёт реальные байты |
| Cobalt-инстансы на «плохих» IP | ❌ `youtube.login` или пустой туннель |

**Вывод:** проблема не в инструменте, не в фингерпринте плеер-клиента и не в
PO-токене. Блокируется **egress-IP**. Смена клиента, апгрейд yt-dlp и свой Cobalt
на этом же сервере упираются в один и тот же IP и не дают ничего.

> Любое рабочее решение обязано менять точку выхода в интернет. Это главный
> критерий, по которому отсортирован список ниже.

## 2. Про Turnstile

Turnstile — это анти-абуз **публичных Cobalt-инстансов**, а не YouTube. Их
владельцы защищаются от таких нагрузок, как наша.

YouTube никакого Turnstile не использует: у него собственная проверка репутации
IP/сессии («Sign in to confirm»). Это не капча, которую решают, а гейт, который
либо пропускает IP, либо нет.

Следствие: **как только мы уходим с чужих инстансов на свой, Turnstile исчезает
из уравнения полностью** — в self-hosted Cobalt он не включается, пока не
прописаны ключи. Остаётся ровно один гейт: репутация IP.

## 3. Решения, по убыванию рекомендации

### 1. Резидентный прокси только на скачивание + вернуть yt-dlp ⭐

Backend качает сам, трафик к YouTube идёт через резидентный/мобильный IP. Ни
чужих инстансов, ни Turnstile, ни лотереи.

- **Стоимость:** $1–3/ГБ. Аудио 20-минутного ролика ≈ 25 МБ → **$0.03–0.08 за
  видео**. Через прокси гоняется только скачивание, не весь сервер.
- **Работы:** вернуть `yt_dlp` в `download_youtube.py` с `--proxy`, Cobalt
  оставить вторым эшелоном.
- **Надёжность:** высокая и предсказуемая — единственный вариант, где мы
  управляем обеими переменными.
- **Минус:** переменная статья расходов; качать `bestaudio`, не видео.

### 2. Свой Cobalt + тот же прокси

То же самое, но сохраняется текущая архитектура и бесплатно достаётся
TikTok/Instagram.

- **Стоимость:** прокси + ~$5/мес контейнер (можно на том же CX43).
- **Работы:** больше, чем в п.1 (docker + конфиг + прокси внутрь контейнера),
  и мы становимся мейнтейнером Cobalt.
- **Когда выбирать:** если поддержка не-YouTube сервисов важнее простоты.
  yt-dlp их тоже умеет, поэтому чаще выигрывает п.1.

### 3. Нода скачивания на «жилом» интернете

Мини-сервер или Raspberry Pi в офисе/дома, backend ходит к нему по HTTP.
Домашний IP по определению резидентный.

- **Стоимость:** ~$0/мес сверх существующего интернета.
- **Минус:** домашний IP под нагрузкой, зависимость от аптайма квартиры, нужен
  туннель (Tailscale / Cloudflare Tunnel). Хороший дешёвый старт, плохая
  единственная опора продакшена.

### 4. Платный сторонний download/transcript API

Отдать проблему вендору целиком.

- **Стоимость:** обычно дороже прокси на единицу, но ноль эксплуатации.
- **Минус:** вендор-лок, тот же риск блокировок (просто не наш), деньги за
  каждый запрос.
- **Когда выбирать:** если время дороже денег и объёмы небольшие.

### 5. Ротация публичных Cobalt (текущее состояние)

Ротация починена 2026-07-24 (коммит `2039aec`): перебор идёт и по этапу
скачивания, а не только резолва. Оставить как **резервный слой**, не как основной.

- **Стоимость:** $0. **Надёжность:** лотерея. На 2026-07-24 из шести известных
  инстансов живым был **один**.

### 6. Cookies + PO-token от реального аккаунта ⚠️

- Токены привязаны к video ID, ручное извлечение больше не работает — нужен
  провайдер PO-токенов отдельным сервисом.
- По текущим отчётам PO-токен **уже не снимает бот-чек** в большинстве случаев
  на датацентровом IP, то есть не лечит первопричину.
- Куки живого аккаунта на заблокированном IP — прямой путь к бану аккаунта.
- Годится **только как усилитель поверх п.1**, никогда самостоятельно.

### 7. Смена IP/провайдера без резидентности ❌

Новый IP у Hetzner, другой регион, AWS/GCP — репутация считается по ASN,
датацентровые диапазоны флагуются целыми блоками.

### 8. YouTube Data API v3 ❌

Официальный API **не отдаёт** ни аудио, ни видео, а `captions.download` работает
только для собственных роликов. Легального API под нашу задачу у YouTube нет.

## 4. План

- **Сейчас:** добавить `https://api.cobalt.liubquanti.click/` в админ-панель —
  это единственный живой инстанс. Важно: список в панели
  (`data/cobalt-config.json`) **перекрывает** встроенные дефолты из
  `youtube_cobalt.py` (см. `getEffectiveCobaltUrls`), поэтому сейчас рабочий
  инстанс просто отключён. Это заплатка на сегодня, не решение.
- **На неделю:** п.1 — pay-as-you-go резидентный прокси без минималки, вернуть
  yt-dlp с `--proxy`, Cobalt-ротацию оставить фолбэком. Два независимых пути
  вместо одного ненадёжного.
- **Дальше:** при росте объёмов считать `$ / успешное скачивание`, а не цену за
  ГБ, и по этой метрике выбирать между прокси и вендором.

## 5. Открытые вопросы и риски

- **Не проверено эмпирически:** резидентный прокси. Нужен оплаченный пробный ГБ,
  дальше сквозной тест на сервере занимает пару минут.
- **Смежная проблема:** хардкод-таймаут 120 с в `src/services/youtubeService.ts`
  (`downloadMediaAudio`) не укладывается на длинном видео + медленном релее.
- **ToS:** скачивание с YouTube вне официального API нарушает его условия. Это
  риск уровня продукта, а не инженерии, и решение принимается осознанно.
